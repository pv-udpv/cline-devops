# Global Cline Development Standards

## Context Awareness
- Always check project structure before making changes
- Use `tree` or `ls -R` to understand layout
- Identify critical files: pyproject.toml, .env, README.md

## Python Standards (3.12+)
- Package manager: `uv`
- Always create `.venv`: `uv venv`
- Install before import: `uv add <package>`
- Type hints mandatory for public APIs
- Pydantic for data validation

## Workflow
1. **Analysis Phase**: Understand task, check dependencies
2. **Planning Phase**: Break into steps, estimate context usage
3. **Execution Phase**: Implement with tests
4. **Validation Phase**: Run tests, check types

## Context Management
- Max context: 200K tokens (Claude Sonnet)
- Use file summaries for large codebases
- Prefer targeted file reads over full project scans

## Testing
- Write tests alongside code
- Use pytest with markers: @pytest.mark.unit, @pytest.mark.integration
- Generate test data with Faker/Factory

## Documentation
- Docstrings: Google style
- README.md: Problem, Solution, Usage, API
- CHANGELOG.md: Keep-a-Changelog format

## Error Handling
- Use custom exceptions hierarchy
- Log with structlog
- Never catch generic Exception without re-raising

## CLI Tools
- Framework: Typer
- Config: YAML/TOML with validation
- Help text: comprehensive and with examples

## Git Workflow
- Branch naming: `feature/`, `fix/`, `refactor/`
- Commits: Conventional Commits format
- Pre-commit: ruff, mypy, pytest

## MCP Integration
- Check for available MCP servers before implementing
- Prefer official MCP over custom implementations
- Document MCP dependencies in README
